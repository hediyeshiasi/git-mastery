Part 2 — How Git Works

درک نحوه عملکرد Git برای استفاده صحیح و حرفه‌ای از آن ضروری است. Git یک سیستم کنترل نسخه توزیع‌شده (Distributed Version Control System) است که تغییرات پروژه را به‌صورت ساختاریافته و قابل ردیابی مدیریت می‌کند.

معماری کلی Git

Git از سه بخش اصلی تشکیل شده است:

1️⃣ Working Directory

محیط کاری فعلی پروژه است که فایل‌ها در آن قرار دارند و تغییرات ابتدا در این بخش ایجاد می‌شوند.

2️⃣ Staging Area (Index)

بخشی میانی که تغییرات انتخاب‌شده برای ثبت نهایی در آن قرار می‌گیرند.

دستور git add تغییرات را از Working Directory به Staging Area منتقل می‌کند.

3️⃣ Repository

محل ذخیره تاریخچه کامل پروژه است.

با اجرای دستور git commit، یک Snapshot جدید در Repository ثبت می‌شود.

نحوه ثبت تغییرات در Git

فرآیند ثبت تغییرات در Git به صورت مرحله‌ای انجام می‌شود:

  1. ایجاد یا ویرایش فایل در Working Directory
  2. انتقال تغییرات به Staging Area
  3. ثبت نهایی تغییرات در Repository به‌صورت یک Commit

هر Commit شامل یک Snapshot از وضعیت پروژه در همان لحظه است.

Snapshot در Git

تعریف Snapshot

در Git، Snapshot نشان‌دهنده وضعیت کامل پروژه در یک لحظه مشخص است. اما این وضعیت به‌صورت کپی کامل از همه فایل‌ها ذخیره نمی‌شود.

Git تنها تغییرات واقعی را ذخیره می‌کند و برای فایل‌های بدون تغییر، به نسخه‌های قبلی اشاره می‌کند.

نحوه عملکرد Snapshot

هنگام اجرای git commit:

  • Git بررسی می‌کند کدام فایل‌ها تغییر کرده‌اند.
  • برای فایل‌های تغییر کرده، نسخه جدید ذخیره می‌شود.
  • برای فایل‌های بدون تغییر، یک اشاره (Reference) به نسخه قبلی ثبت می‌شود.
  • مجموعه این اطلاعات، Snapshot جدید را تشکیل می‌دهد.

بنابراین Snapshot شامل داده‌های زیر است:

  • اشاره به نسخه‌های مشخص فایل‌ها
  • اطلاعات متادیتا (نویسنده، زمان ثبت، پیام Commit)
  • اشاره به Snapshot قبلی

ساختار داخلی Commit

  • یک شناسه یکتا (Hash)
  • مرجع به Snapshot قبلی
  • اطلاعات مربوط به نویسنده
  • زمان ثبت
  • پیام توضیحی

شناسه یکتا بر اساس محتوای Commit تولید می‌شود و تضمین می‌کند هر نسخه کاملاً قابل ردیابی و غیرقابل تداخل باشد.

بازسازی نسخه‌های قبلی

  • Git به Snapshot مربوطه مراجعه می‌کند.
  • وضعیت دقیق فایل‌ها بر اساس اشاره‌های ثبت‌شده بازسازی می‌شود.
  • پروژه دقیقاً به همان حالت ذخیره‌شده برمی‌گردد.

این فرآیند بدون بازنویسی غیرضروری فایل‌ها و بر اساس ساختار ارجاعی داخلی Git انجام می‌شود.

ویژگی‌های کلیدی مدل Snapshot

  • ذخیره‌سازی بهینه بر اساس تغییرات واقعی
  • جلوگیری از کپی‌های غیرضروری
  • امکان ردیابی دقیق تاریخچه
  • قابلیت بازسازی کامل هر نسخه
  • ارتباط زنجیره‌ای بین Commitها

جمع‌بندی

  • Git دارای سه بخش اصلی است: Working Directory، Staging Area و Repository
  • ثبت تغییرات به‌صورت مرحله‌ای انجام می‌شود.
  • هر Commit شامل یک Snapshot از وضعیت پروژه است.
  • Snapshot بر پایه ذخیره تغییرات واقعی و ارجاع به نسخه‌های قبلی ساخته می‌شود.
  • هر Commit دارای شناسه یکتا و قابل ردیابی است.
  • امکان بازسازی دقیق هر نسخه در هر زمان وجود دارد.

درک این ساختار، پایه استفاده حرفه‌ای از Git محسوب می‌شود.
