09 — Collaboration & Workflow

Git برای همکاری طراحی شده است.

در این فصل یاد می‌گیریم چگونه چند توسعه‌دهنده به‌صورت همزمان روی یک پروژه کار می‌کنند بدون اینکه تاریخچه تخریب شود.

Pull Request چیست؟

Pull Request (PR) مکانیزمی در پلتفرم‌هایی مثل GitHub است.

هدف آن:

  • پیشنهاد ادغام یک Branch
  • بررسی کد (Code Review)
  • بحث و اصلاح قبل از Merge

Pull Request خودِ Git نیست.
ویژگی‌ای در سرویس‌های میزبانی Git است.

فرآیند استاندارد همکاری

  1. Pull آخرین تغییرات از main
  2. ساخت Branch جدید
  3. توسعه Feature
  4. Push Branch
  5. ایجاد Pull Request
  6. Code Review
  7. Merge به main
  8. حذف Branch

این الگو در اکثر تیم‌های حرفه‌ای استفاده می‌شود.

Code Review

قبل از Merge:

  • کد بررسی می‌شود
  • استانداردها چک می‌شود
  • باگ‌ها شناسایی می‌شود
  • معماری ارزیابی می‌شود

هدف Review:
افزایش کیفیت و کاهش ریسک.

استراتژی‌های Workflow

1️⃣ Git Flow

دارای Branchهای مشخص:

  • main (نسخه پایدار)
  • develop (توسعه فعال)
  • feature/*
  • release/*
  • hotfix/*

مناسب پروژه‌های بزرگ با چرخه انتشار رسمی.

2️⃣ Trunk Based Development

فقط یک Branch اصلی (معمولاً main)

Featureها کوتاه‌مدت هستند و سریع Merge می‌شوند.

مناسب تیم‌های سریع و Continuous Deployment.

3️⃣ Feature Branch Workflow

هر Feature در Branch جدا
پس از تکمیل → Merge به main

ساده، رایج و قابل مدیریت.

Release Strategy

برای نسخه‌دهی از Tag استفاده می‌شود:

git tag v1.0.0
git push origin v1.0.0

Tag یک اشاره‌گر ثابت به یک Commit مشخص است.

برای نسخه‌های پایدار استفاده می‌شود.

Semantic Versioning

الگوی رایج نسخه‌دهی:

MAJOR.MINOR.PATCH

مثال:

1.4.2

  • MAJOR → تغییرات ناسازگار
  • MINOR → افزودن قابلیت
  • PATCH → اصلاح باگ

اصول حرفه‌ای همکاری

  • Commitهای کوچک و هدفمند
  • Branchهای کوتاه‌مدت
  • Pull Request اجباری
  • عدم Force Push روی Branch مشترک
  • تاریخچه خوانا و معنادار

Consistency مهم‌تر از پیچیدگی Workflow است.

مدل ذهنی نهایی

Git ابزار ذخیره‌سازی نیست.
سیستم مدیریت تاریخچه برای همکاری است.

  • Branch → ایزوله‌سازی
  • Merge → ترکیب
  • Remote → اشتراک
  • Pull Request → کنترل کیفیت

این چهار ستون همکاری حرفه‌ای در Git هستند.

نتیجه

اکنون می‌توانید:

  • در تیم کار کنید
  • Pull Request ایجاد کنید
  • Workflow مناسب انتخاب کنید
  • نسخه منتشر کنید
  • تاریخچه تمیز نگه دارید

در فصل بعد به جمع‌بندی نهایی و طراحی Workflow عملی پروژه می‌رسیم.
