03 — Staging and Commit

مقدمه

پس از ایجاد Repository، Git آماده مدیریت نسخه‌ها است. اما تا زمانی که تغییرات را به‌صورت رسمی ثبت نکنیم، هیچ سابقه‌ای در تاریخچه پروژه ایجاد نمی‌شود.

در این بخش با دو مفهوم کلیدی آشنا می‌شویم:

  • Staging Area
  • Commit (Snapshot)

این دو مفهوم، هسته اصلی سیستم نسخه‌بندی در Git را تشکیل می‌دهند.

مدل سه‌مرحله‌ای Git

Git تغییرات را در سه ناحیه جداگانه مدیریت می‌کند:

  1. Working Directory محل واقعی فایل‌ها روی سیستم شما؛ جایی که ایجاد و ویرایش انجام می‌شود.
  2. Staging Area (Index) ناحیه‌ای واسط برای انتخاب تغییراتی که قرار است ثبت شوند.
  3. Repository محل ذخیره دائمی Snapshotها و تاریخچه پروژه.

جریان ثبت تغییرات:

Working Directory → Staging Area → Repository

هیچ تغییری بدون دستور صریح شما وارد Repository نمی‌شود.

بررسی وضعیت پروژه

برای مشاهده وضعیت فایل‌ها از دستور زیر استفاده می‌کنیم:

git status

این دستور:

  • وضعیت فایل‌ها را گزارش می‌دهد
  • نشان می‌دهد چه فایل‌هایی جدید هستند
  • مشخص می‌کند کدام فایل‌ها تغییر کرده‌اند
  • نمایش می‌دهد چه فایل‌هایی آماده Commit هستند

مفهوم Untracked

اگر فایلی ایجاد شود اما هنوز به Git معرفی نشده باشد، در وضعیت Untracked قرار دارد.

Untracked files:
  note.txt

یعنی Git فایل را می‌بیند، اما هنوز آن را برای ثبت انتخاب نکرده‌اید.

ورود فایل به Staging Area

برای انتخاب یک فایل جهت ثبت:

git add note.txt

اکنون فایل وارد Staging Area شده است.

بررسی مجدد وضعیت:

git status
Changes to be committed:
  new file: note.txt

در این مرحله فایل آماده ثبت در تاریخچه است، اما هنوز Commit انجام نشده است.

Commit چیست؟

Commit یک Snapshot از وضعیت پروژه در یک لحظه مشخص است.

هنگامی که Commit انجام می‌شود:

  • محتوای فایل‌های موجود در Staging Area ذخیره می‌شود
  • یک شناسه یکتا (Hash) تولید می‌شود
  • یک نقطه جدید در تاریخچه پروژه ایجاد می‌شود

هر Commit نمایانگر یک نسخه مشخص و قابل بازگشت از پروژه است.

ایجاد Commit

برای ثبت تغییرات آماده‌شده:

git commit -m "Add initial note file"

پارامتر -m برای تعیین پیام Commit استفاده می‌شود.

پیام Commit باید:

  • کوتاه
  • شفاف
  • توصیف‌کننده تغییر انجام‌شده

پس از اجرای این دستور، Snapshot ساخته شده و وارد Repository می‌شود.

مشاهده تاریخچه Commitها

برای مشاهده تاریخچه:

git log --oneline
a1b2c3d Add initial note file

هر Commit دارای یک شناسه یکتای ۴۰ کاراکتری است؛ نمایش کوتاه آن برای شناسایی کافی است.

وضعیت‌های مختلف فایل

وضعیت توضیح
Untracked فایل جدیدی که هنوز به Git معرفی نشده
Modified فایل ثبت‌شده‌ای که تغییر کرده اما Staged نشده
Staged فایل آماده ثبت در Commit
Committed فایل ثبت‌شده در تاریخچه

چرخه استاندارد ثبت تغییرات

فرآیند معمول:

  • ایجاد یا ویرایش فایل
  • بررسی وضعیت با git status
  • انتخاب فایل با git add
  • ثبت نهایی با git commit

این چرخه در تمام پروژه‌های Git تکرار می‌شود.

نکته مفهومی مهم

Git هیچ‌گاه تغییرات را به‌صورت خودکار ذخیره نمی‌کند.

این طراحی باعث می‌شود:

  • تاریخچه پروژه تمیز بماند
  • فقط تغییرات مهم ثبت شوند
  • کنترل کامل در اختیار توسعه‌دهنده باشد

جمع‌بندی

در این بخش آموختیم:

  • Staging Area چیست
  • چگونه فایل‌ها را برای ثبت انتخاب کنیم
  • Commit چگونه Snapshot ایجاد می‌کند
  • ساختار سه‌مرحله‌ای Git چگونه کار می‌کند

از این نقطه به بعد، پروژه شما دارای تاریخچه نسخه‌ها است و می‌توان تغییرات را مدیریت و ردیابی کرد.
