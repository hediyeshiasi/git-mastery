04 — Git Status Explained

مقدمه

دستور git status یکی از پرکاربردترین و مهم‌ترین دستورات در Git است. تقریباً قبل از هر Commit، توسعه‌دهندگان این دستور را اجرا می‌کنند تا ببینند:

  • چه فایل‌هایی تغییر کرده‌اند
  • کدام فایل‌ها Stage شده‌اند
  • چه فایل‌هایی هنوز توسط Git دنبال نمی‌شوند
  • وضعیت فعلی شاخه (Branch) چیست

در واقع، git status داشبورد لحظه‌ای پروژه شماست.

چرا git status مهم است؟

Git به‌صورت خودکار تغییرات را ثبت نمی‌کند. بنابراین باید همیشه بدانیم پروژه در چه وضعیتی قرار دارد.

git status به ما کمک می‌کند:

  • از وضعیت دقیق فایل‌ها آگاه شویم
  • قبل از Commit اشتباه نکنیم
  • تاریخچه تمیز و کنترل‌شده بسازیم

ساختار کلی خروجی git status

خروجی این دستور معمولاً شامل بخش‌های زیر است:

  1. وضعیت Branch
  2. فایل‌های Stage شده
  3. فایل‌های تغییر کرده ولی Stage نشده
  4. فایل‌های Untracked

نمونه خروجی ساده

git status
On branch main

Changes to be committed:
  modified:   app.js

Changes not staged for commit:
  modified:   style.css

Untracked files:
  test.txt

بیایید این خروجی را تحلیل کنیم.

بخش اول: وضعیت Branch

On branch main

این خط نشان می‌دهد که در حال حاضر روی چه شاخه‌ای کار می‌کنید. در اغلب پروژه‌های جدید، شاخه پیش‌فرض main است.

بخش دوم: Changes to be committed

Changes to be committed:
  modified:   app.js

این فایل‌ها در Staging Area قرار دارند، یعنی:

  • با git add انتخاب شده‌اند
  • آماده ثبت در Commit هستند
  • در Commit بعدی ذخیره خواهند شد

بخش سوم: Changes not staged for commit

Changes not staged for commit:
  modified:   style.css

این فایل‌ها:

  • قبلاً توسط Git شناخته شده‌اند
  • تغییر کرده‌اند
  • اما هنوز وارد Staging Area نشده‌اند

برای Stage کردن آن‌ها:

git add style.css

بخش چهارم: Untracked files

Untracked files:
  test.txt

این فایل‌ها:

  • جدید هستند
  • هنوز توسط Git دنبال نمی‌شوند
  • در هیچ Commitی وجود ندارند

برای معرفی آن‌ها به Git:

git add test.txt

دیاگرام وضعیت فایل‌ها

فرآیند حرکت فایل‌ها بین وضعیت‌های مختلف:


Untracked
   │
   ▼
Tracked (بعد از git add)
   │
   ▼
Staged
   │
   ▼
Committed

اگر فایلی که قبلاً Commit شده تغییر کند:


Committed
   │
   ▼
Modified
   │
   ├── git add → Staged
   └── بدون add → باقی در Modified

جدول کامل وضعیت‌ها و نیاز به git add

وضعیت توضیح نیاز به git add؟
Untracked فایل جدیدبله
Modified فایل قبلی که تغییر کرده بله
Staged آماده Commit خیر
Committed ثبت‌شده در تاریخچه خیر

نکات حرفه‌ای

۱) اجرای مداوم git status

بهتر است قبل از هر Commit این دستور را اجرا کنید تا مطمئن شوید:

  • فقط فایل‌های مورد نظر Stage شده‌اند
  • فایل ناخواسته‌ای وارد Commit نمی‌شود

۲) درک پیام‌های راهنما

Git در خروجی git status معمولاً پیشنهاد می‌دهد: 

(use "git add <file>..." to update what will be committed)

این پیام‌ها بسیار کاربردی هستند و به شما می‌گویند قدم بعدی چیست.

۳) هیچ تغییری بدون گزارش باقی نمی‌ماند

یکی از مزایای مهم Git این است که تا زمانی که Commit نکنید، همیشه می‌توانید ببینید چه چیزی تغییر کرده است.

git status ابزار اصلی این شفافیت است.

جمع‌بندی

در این بخش یاد گرفتیم:

  • git status چه اطلاعاتی نمایش می‌دهد
  • تفاوت بین Untracked، Modified و Staged چیست
  • چگونه فایل‌ها بین وضعیت‌های مختلف حرکت می‌کنند
  • چرا این دستور پایه کنترل پروژه است

اگر Git را یک سیستم نسخه‌بندی قدرتمند بدانیم، git status چراغ راهنمای شما در تمام مسیر توسعه خواهد بود.

در بخش بعدی به بررسی عمیق‌تر تاریخچه پروژه با git log می‌پردازیم.
